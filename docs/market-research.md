# 市场调研与方向选择

调研目标：为“AI 小说转剧本工具”选择最适合七牛 XEngineer 题目与评审规则的产品切口。

## 结论

本项目选择“中文小说/IP 到短剧化分场剧本 YAML 初稿”作为核心方向，而不是通用 AI 编剧平台。

原因：

- 中文网文和短剧改编场景更贴近真实内容生产需求。
- 题目硬性要求 YAML Schema，结构化中间层比单纯聊天生成更能体现工程能力。
- 竞品多集中在写作辅助、剧本编辑或灵感生成，较少强调“3 章以上小说输入、来源追踪、Schema 校验、可编辑 YAML 输出”这一组合。
- 七牛 XEngineer 强调产品架构师能力，该方向能同时展示需求判断、产品设计、架构拆分和代码实现。

## 竞品观察

### 通用 AI 写作工具

代表：Sudowrite、Squibler 等。

特点：

- 强在灵感生成、文本续写、段落润色。
- 面向作者写作过程，而不是面向改编流水线。
- 输出通常是自然语言文本，结构化约束较弱。

启发：

- 本项目应避免只做“生成一段剧本”的聊天框。
- 应强调章节理解、结构拆分、Schema 校验和可编辑输出。

### AI 编剧与剧本工具

代表：NolanAI、Celtx 相关创作工具、Dramatron 研究项目。

特点：

- 更贴近剧本创作和编剧协作。
- 通常围绕故事大纲、角色、对白、场景展开。
- 更关注创作辅助，而不是从长篇小说章节自动改编。

启发：

- 本项目的剧本结构应包含角色、场景、动作、对白、旁白和转场。
- 可以借鉴专业剧本元素，但不做完整专业剧本编辑器。

### 剧本格式与结构化输出

代表：Fountain 纯文本剧本格式、Final Draft 等专业工具生态。

特点：

- 强调场景、角色、对白、动作等剧本元素。
- 传统格式适合阅读和排版，但不一定适合作为 AI 流水线中间层。

启发：

- YAML 更适合作为机器可校验、可编辑、可扩展的中间层。
- Schema 中应保留 `characters`、`episodes`、`scenes`、`elements` 等层级。

## 用户与场景

目标用户：

- 网文作者：希望快速看到作品的短剧改编初稿。
- 短剧编剧：希望从小说章节中快速获得场次、冲突和对白草稿。
- 内容团队：希望评估 IP 是否适合短剧化。

核心任务：

1. 输入 3 章以上小说文本。
2. 自动识别章节、角色、地点和关键事件。
3. 生成短剧分场剧本。
4. 输出可编辑 YAML。
5. 校验 YAML 是否满足 Schema。

## 产品方向取舍

### 方向 A：通用 AI 编剧平台

优点：想象空间大。

缺点：范围过大，容易变成普通聊天生成器；很难在短时间内体现工程深度。

结论：不采用。

### 方向 B：传统影视剧本格式转换器

优点：剧本格式标准更成熟。

缺点：与中文网文、短剧改编的市场热度和题目差异化不够贴合。

结论：不作为 MVP 核心。

### 方向 C：小说到短剧 YAML 转换工具

优点：题目匹配度高；市场场景明确；便于展示产品、架构、测试和创新点。

缺点：mock 模式下改编质量不等同真实大模型效果，需要清楚说明 API provider 可扩展。

结论：采用。

## 差异化设计

- `source_chapters`：每个场景保留来源章节，证明不是凭空生成。
- `adaptation_note`：说明小说叙述如何被改写为短剧场景。
- `hook`：突出短剧结尾钩子。
- Schema 校验：让 YAML 不只是展示文本，而是可验证的数据结构。
- mock/API 双模式：无 Key 可演示，有 Key 可扩展。
